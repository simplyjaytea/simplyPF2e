# simplyPF2e — condensed project brief

Foundry VTT module. AI generates PF2e (Remaster) NPC/monster concepts; module computes numbers from GM Core benchmark tables; compendium matching fills real items. Repo: `simplyjaytea/simplyPF2e`, GitHub.

## Pipeline (per creature)

1. `generateConcept()` (scripts/ai.mjs) — huge SYSTEM_PROMPT, one AI call, returns JSON concept (scales not numbers: str/dex/etc, AC/HP/save scales, strikes, specialAbilities, feats, equipment, loot, spellcasting draft).
2. `normalizeConcept()` (builder.mjs) — coerce/clamp raw JSON into safe shape.
3. Spells (if spellcasting): `chooseSpellFocus()` — tiny call, concept only, returns 3-6 keyword tags. Then `getSpellCandidates(tradition, maxRank, keywords)` (compendium.mjs) — pulls real tradition spells, filters by keyword match on traits/name, falls back to full list if filtered <12. Then `selectSpells()` — final AI pick from that narrowed list. 3 AI calls total for casters.
4. Equipment (if any): `#refineEquipment()` (generator-app.mjs) — keywords tokenized locally from first-draft gear + strike names (no AI focus call), `getEquipmentCandidates(level, keywords)` (compendium.mjs) pulls real level-capped non-treasure items, `selectEquipment()` picks final gear. 1 extra AI call; any failure keeps first-draft names.
5. `resolveConcept()` (builder.mjs) — fuzzy-match abilities/feats/equipment/loot names against compendium via `findEntry()`.
6. `createActor()` (builder.mjs) — build real Foundry items, apply runes/quantities/carry state, create actor.

Encounter mode: `designEncounter()` picks theme+briefs once, then full per-creature pipeline runs once per party member (N× system prompt cost).

Loot reroll: `generateLoot()` — separate small call, concept summary only (blurb/level/rarity/traits, no full description), regenerates just loot array.

## Key files

- `scripts/ai.mjs` — all AI calls + SYSTEM_PROMPT (creature schema) + LOOT_GUIDE. Streaming SSE, retry-once, JSON repair for truncation.
- `scripts/builder.mjs` — normalizeConcept, resolveConcept, computeStats (GM Core table lookups via tables.mjs), createActor, enrichDescription (text→clickable roll links).
- `scripts/compendium.mjs` — findEntry fuzzy match, getSpellCandidates, getPacksFor (configurable source packs per category).
- `scripts/generator-app.mjs` — UI app, orchestrates pipeline, token usage tracking (#recordTokens), progress steps.
- `scripts/tables.mjs` — GM Core Building Creatures benchmark tables.
- `scripts/encounter.mjs` — XP budget/composition math.
- `scripts/rule-templates.mjs` — item forge: harvests real Rule Element exemplars from installed compendiums (never hand-authors RE JSON).
- `scripts/item-builder.mjs` — item forge: concept normalize, empirical pricing, item data assembly from cloned exemplar rules (wondrous items) or from real base-item + rune-item documents summed (runed weapons/armor, Phase 3).
- `scripts/itemforge-app.mjs` — item forge UI (Items directory button), mirrors generator-app patterns.

- Magic item forge Phase 1 (branch `feat/magic-item-forge`): describe a wondrous item → real Item document with WORKING passive Rule Elements, button on the Items directory (`.spf-itemforge-button`, `module.api.openItemForge`). **Safety principle (reusable pattern): clone real REs, never hand-author them** — Foundry fails silently on a wrong RE key/field, so `rule-templates.mjs` scans installed Item packs at runtime (custom-field index `{fields:["system.rules"]}`, getDocuments fallback, cached) for published items already carrying each RE kind, exact-shape whitelists the rule (no predicate/label/mode baggage; string type, numeric value), and `buildMagicItemData()` clones+parameterizes only value/selector/damage-type. 6 effect kinds all shipped with runtime-found exemplars: itemBonus→FlatModifier(type item), resistance→Resistance, weakness→Weakness, immunity→Immunity, sense→Sense, **speed→BaseSpeed (kept, NOT dropped** — verified real numeric-valued equipment exemplars exist, e.g. Swooping Wings `{key:"BaseSpeed",selector:"fly",value:25}`; formula-valued ones like Ring of Swimming are filtered out). Field shapes verified against foundryvtt/pf2e pack source (hellfire-boots, choral-toga, eyes-of-the-cat, diviners-nose-chain, skeleton-blaze). Kinds with no exemplar in a given world are dropped from the AI's schema dynamically (prompt enum built from `availableEffectKinds()`), warned in console + UI, never hand-authored as fallback. Weakness's default-world exemplar likely comes from the bestiary family glossary (equipment Weakness rules are often array-typed and rejected) — scan order equipment→feats→abilities→all Item packs handles it. Pricing is empirical too: `priceForLevel()` = median real compendium gp at the item level (`EQUIPMENT_TYPES` minus treasure, `priceToGp()`), window widens ±1,±2,... when <5 samples (documented fallback, no memorized price table), × `RARITY_TREASURE_MULTIPLIER`. Usage strings are harvested from real magical equipment (`getUsageOptions()`, they're concatenated slugs like "wornshoes" — never invent the format). `generateMagicItemConcept()` (ai.mjs) is one JSON call via the shared `requestJSON` plumbing. Res/weak values clamp to the GM Core RESISTANCE table max for the level.

- Magic item forge Phase 3 (branch `feat/itemforge-phase3`, builds on Phase 1/2): **rune-based magic weapons/armor**, a "kind" select on the item forge (Wondrous Item / Weapon / Armor) switches to a wholly different, simpler pipeline. **Key discovery that shaped the design**: fundamental AND property runes are just real Item documents in the equipment compendium (type `"equipment"`, e.g. "Weapon Potency (+1)", "Striking (Greater)", "Flaming") with their own real level and price — verified by pulling the actual foundryvtt/pf2e system source from GitHub (`packs/pf2e/equipment/*.json`, `src/module/item/weapon/data.ts`, `src/module/item/armor/data.ts`), not recalled from memory. So a runed item needs **no empirical price benchmark and no memorized rune list** (unlike Phase 1's wondrous items): pick a real base weapon/armor plus real rune items and SUM their real prices; the item's overall level is the MAX level among them (the actual PF2e rule) — computed by `buildRunedItemData()` at generation time, so for Phase 3 the "preview" IS the final item data (no separate build-at-create step like the wondrous/activation paths). **Candidate harvesting** (`getBaseItemCandidates`, `getPropertyRuneCandidates`, `getFundamentalRuneTiers` in item-builder.mjs) scans the equipment-pack index once (`getAllEquipmentEntries`, cached) and filters: base items by `type === "weapon"|"armor"` and level; property runes by their real `system.usage.value` string (`"etched-onto-a-weapon"` for weapons; `"etched-onto-armor"`/`"etched-onto-light-armor"`/`"etched-onto-med-heavy-armor"` for armor — verified against real rune items, e.g. Ghost Touch, Flaming, Fortification, Winged); fundamental tiers by resolving each tier's REAL item level against the target (never a hardcoded threshold — Weapon Potency (+1) is level 2, Armor Potency (+1) is level 5, so low-level targets correctly get zero available potency tiers and the app surfaces a clear error instead of the AI call ever running). **The one non-obvious transform**: `system.runes.property` stores rune keys in **camelCase** ("ghostTouch", "greaterFlaming", "ancestralEchoing"), not the kebab-case slug the compendium item's own name would normally produce ("ghost-touch") — verified against `WEAPON_PROPERTY_RUNES`/`ARMOR_PROPERTY_RUNES` in the system's `runes.ts` source. `kebabToCamel(slugify(name))` bridges the two; this was the single highest-risk spot for a silent-failure bug (wrong key = rune data structurally present but mechanically inert, same failure mode Phase 1's whole cloning strategy exists to avoid) so it was confirmed against 4 independent real examples before use, not just one. The AI (`generateRunedItemConcept`, ai.mjs) picks ONLY from real candidate names/tiers it's shown — normalize matches every name back against those lists (`normalizeRunedItemConcept`), dropping anything unmatched with a console warning, same "never invented" discipline as the rest of the module. Final item name follows the standard PF2e "+N [secondary adjective] [property rune names] [base name]" convention, assembled from the resolved real documents' own names — never AI-invented. Rarity bumps to the GM's chosen rarity only if higher than the base item's own; "magical" trait always added. **Known gaps**: see Known gaps section below (no rune prerequisite/exclusivity checking, armor-category filtering is loose, shield/ammo runes out of scope).

## Bug log

- **Attack of Opportunity not generating** (fixed, unreleased). Schema comment (ai.mjs specialAbilities.glossary) cited "Attack of Opportunity" as a valid glossary-ability example, but the design-guidance "use standard glossary abilities" enumeration bullet never listed it, so the model almost never picked it — schema comments alone don't drive output frequency, the design-guidance bullets do. Fix: added "Attack of Opportunity" to that enumeration + explicit call-out on the Soldier road-map line. **Pattern for future "X not generating" bugs**: check whether X is only mentioned once as a schema-comment example vs. actually reinforced in the "Design guidance" section — if it's schema-only, add it to the relevant enumeration/road-map bullet there, same fix shape.
- **Gold coins appearing in equipment** (fixed, unreleased). Coins are only supposed to live in `loot` (handled by `parseCoins()`/`normalizeLoot()` in builder.mjs, which map "Gold Coins"/"150 gold pieces"/"20 gp" to real currency). The `equipment` schema comment never told the model NOT to put coins there, so it sometimes did, and equipment has no coin-to-currency conversion — the coin name just became a junk placeholder item via `customEquipmentItem()`. Two-part fix: (1) added "NO coins or currency here" to the equipment schema comment; (2) defensive code fix reusing existing `parseCoins()` in `normalizeConcept()`'s equipment mapping (builder.mjs) to drop any equipment entry that parses as pure coins, regardless of what the model outputs. **Pattern**: prompt-only fixes are necessary but not sufficient for "wrong bucket" bugs — pair with a cheap defensive filter using logic that already exists elsewhere in the file, so a prompt regression doesn't silently reopen the bug.
- **Pre-Remaster item names leaking through (e.g. "Thunderstone" instead of "Blasting Stone")** — fixed on branch `fix/remaster-item-names` (commit `8b29a94`), not this branch. `LOOT_GUIDE` already told the model to use current Remaster names with an example, but the `equipment` schema comment never had that reminder at all — same "reinforced in one sibling field, missing in the other" shape as the two bugs above. Fix: extracted the reminder into a shared `REMASTER_NOTE` const (ai.mjs) with two concrete examples (Thunderstone→Blasting Stone, Bag of Holding→Spacious Pouch), interpolated into both `LOOT_GUIDE` and the `equipment` schema comment. **Pattern reinforced**: whenever the model names a published item/ability/spell in more than one schema field, grep for where the naming-convention reminder already exists and make sure every sibling field that also names published content gets the same reminder — ideally via a shared const, not copy-pasted text.
- **Passives were inert prose instead of "just working"** — also on branch `fix/remaster-item-names` (commit `8b29a94`). Custom (non-glossary-matched) passive `specialAbilities` only get automation from `enrichDescription()`'s phrasing conventions (damage/save/check/heal/area) — anything else (an aura, a conditional bonus, a persistent tick effect) is just prose the GM has to remember to apply. Real glossary-matched abilities (Regeneration, All-Around Vision, ...) get cloned wholesale from the compendium and carry actual working Rule Elements. Fix (prompt-only, deliberately NOT a new Rule-Element-generation feature — that's a much bigger, riskier feature, left for later if ever): added a Design-guidance bullet nudging the model to prefer a real glossary match for passives specifically, and to phrase any necessarily-custom mechanical passive using the existing DESCRIPTION CONVENTIONS so it stays clickable instead of unactionable prose.

- **Encounter mode created one actor per GROUP, ignoring `member.count`** — fixed on branch `fix/encounter-actor-count-and-loot-scaling`. `composeEncounter()` correctly returns groups with counts (e.g. 4 minions sharing one statblock/concept — one AI concept per group is intentional and unchanged), but `#createEncounterActors()` (generator-app.mjs) called `createActor()` once per member, so "4 goblin skirmishers" produced a single actor. Fix: inner loop over `member.count`, one `createActor()` per copy (deterministic from the same concept/resolved, same as the single-creature path), numbered names (`Name 1`, `Name 2`) only when count > 1, `created` counter incremented per copy so the notification reports true totals; `findBestiaryArt()` stays once-per-group (identical minions share art). **Pattern**: when a data structure carries a `count`/quantity field, grep every consumer to confirm it actually multiplies by it — the composer and the consumer were written in different files and the count silently fell on the floor.
- **Loot never scaled up for "hoard"/"lots of loot" requests** — fixed on same branch. Three independent caps each silently flattened abundance: (1) `LOOT_GUIDE` (ai.mjs) hardcoded "3-8 items" with no scaling instruction — added an explicit exception clause (scale to ~12-20 items with more coin/treasure/magic when the description or GM request calls for a hoard/riches/wealth), baseline unchanged; (2) `generateLoot()`'s user context only passed blurb/traits, not `concept.description`, so hoard language living in the description was invisible to Reroll Loot — added a `Description:` line (deliberately reverses the earlier "generateLoot context dropped full description" token trim; correctness beats the small token saving here); (3) `normalizeLoot()` (builder.mjs) hard-truncated to 12 entries — raised to 24 to fit the new upper guidance with headroom while still bounding runaway output. **Pattern**: a "model won't do X even when asked" bug can have BOTH a prompt cause and a post-processing cause — after fixing the prompt, trace the output through every normalize/clamp step, or the code cap silently undoes the prompt fix (and check the model can even SEE the signal: context trimmed for tokens may have dropped the field the instruction keys on).

- **Treasure group budget multiplied by headcount instead of splitting across it** — fixed same session, on `feat/treasure-budget` before merge. First implementation set `member.treasureBudgetEach` to the GROUP's full target and multiplied by `count` for the readout total — so a 4-copy minion group got 4x the intended group treasure instead of one shared share. Fix: `treasureGroupBudget` (constant per group) ÷ `Math.max(count, 1)` = the actual per-copy `treasureBudgetEach`; `#stepMemberCount` re-splits and re-applies live when the count stepper changes. Caught by review before merge, then independently re-audited (traced count 4→3→2→3→4→5, confirmed group total stays ≈ constant, no divide-by-zero, no rounding drift from repeated re-targeting). **Pattern**: same shape as the encounter-actor-count bug below — a per-group total and a per-copy share are easy to conflate; when a UI stepper can change a count live, check whether the derived value needs to be constant (recompute the split) or already-final (don't touch it again).
- **Condition-activation macro would silently auto-apply on an unreadable save result** — fixed on `feat/magic-item-forge-phase2` after independent review (not part of opus's original build, opus's own report explicitly flagged this API as unverified/best-effort). The code read `outcome?.degreeOfSuccess ?? outcome?.options?.degreeOfSuccess`; if neither field existed (our guessed shape being wrong), `dos` was `undefined`, the `typeof dos === "number"` check failed, and `applies` stayed at its initial `true` — silently turning a save-negates condition into an unconditional hit, with no error and no console warning. Fixed to treat an unreadable result the same as an outright API failure: skip auto-applying, log a console warning, post a manual-adjudication chat message. **Pattern**: when wrapping an uncertain external API in a boolean gate, make sure the "I couldn't read the result" branch defaults to the SAFER/weaker outcome, not the stronger one — silent over-application is a worse failure mode than silent under-application, especially for anything with real safety/game-balance stakes.
