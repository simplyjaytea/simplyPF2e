# simplyPF2e â€” condensed project brief

Foundry VTT module. AI generates PF2e (Remaster) NPC/monster concepts; module computes numbers from GM Core benchmark tables; compendium matching fills real items. Repo: `simplyjaytea/simplyPF2e`, GitHub.

## Pipeline (per creature)

1. `generateConcept()` (scripts/ai.mjs) â€” huge SYSTEM_PROMPT, one AI call, returns JSON concept (scales not numbers: str/dex/etc, AC/HP/save scales, strikes, specialAbilities, feats, equipment, loot, spellcasting draft).
2. `normalizeConcept()` (builder.mjs) â€” coerce/clamp raw JSON into safe shape.
3. Spells (if spellcasting): `chooseSpellFocus()` â€” tiny call, concept only, returns 3-6 keyword tags. Then `getSpellCandidates(tradition, maxRank, keywords)` (compendium.mjs) â€” pulls real tradition spells, filters by keyword match on traits/name, falls back to full list if filtered <12. Then `selectSpells()` â€” final AI pick from that narrowed list. 3 AI calls total for casters.
4. Equipment (if any): `#refineEquipment()` (generator-app.mjs) â€” keywords tokenized locally from first-draft gear + strike names (no AI focus call), `getEquipmentCandidates(level, keywords)` (compendium.mjs) pulls real level-capped non-treasure items, `selectEquipment()` picks final gear. 1 extra AI call; any failure keeps first-draft names.
5. `resolveConcept()` (builder.mjs) â€” fuzzy-match abilities/feats/equipment/loot names against compendium via `findEntry()`.
6. `createActor()` (builder.mjs) â€” build real Foundry items, apply runes/quantities/carry state, create actor.

Encounter mode: `designEncounter()` picks theme+briefs once, then full per-creature pipeline runs once per party member (NÃ— system prompt cost).

Loot reroll: `generateLoot()` â€” separate small call, concept summary only (blurb/level/rarity/traits, no full description), regenerates just loot array.

## Key files

- `scripts/ai.mjs` â€” all AI calls + SYSTEM_PROMPT (creature schema) + LOOT_GUIDE. Streaming SSE, retry-once, JSON repair for truncation.
- `scripts/builder.mjs` â€” normalizeConcept, resolveConcept, computeStats (GM Core table lookups via tables.mjs), createActor, enrichDescription (textâ†’clickable roll links).
- `scripts/compendium.mjs` â€” findEntry fuzzy match, getSpellCandidates, getPacksFor (configurable source packs per category).
- `scripts/generator-app.mjs` â€” UI app, orchestrates pipeline, token usage tracking (#recordTokens), progress steps.
- `scripts/tables.mjs` â€” GM Core Building Creatures benchmark tables.
- `scripts/encounter.mjs` â€” XP budget/composition math.
- `scripts/rule-templates.mjs` â€” item forge: harvests real Rule Element exemplars from installed compendiums (never hand-authors RE JSON).
- `scripts/item-builder.mjs` â€” item forge: concept normalize, empirical pricing, item data assembly from cloned exemplar rules (wondrous items) or from real base-item + rune-item documents summed (runed weapons/armor, Phase 3).
- `scripts/itemforge-app.mjs` â€” item forge UI (Items directory button), mirrors generator-app patterns.
- `scripts/pc-builder.mjs` â€” Player Character mode: normalizePCConcept, ABC/grant/feat-slot resolution against the compendium, createCharacterActor (embeds real ancestry/background/class/feat items + sets `system.build` so the pf2e system computes AC/HP/saves itself â€” NPCs get none of this, see below).
- `scripts/pc-tables.mjs` â€” Core Rulebook PC leveling cadence (ability boost / skill increase / ancestry-class-skill-general feat slot levels), hardcoded like tables.mjs hardcodes GM Core's NPC benchmarks. No stat-computation tables here â€” PCs don't need them.

- **Auto-release: pushed tag never triggered the release build** â€” found by testing the auto-release workflow live immediately after merging it (not caught by review, since the YAML is syntactically fine and the logic "push a tag â†’ tag-push trigger fires" is the obviously-correct mental model â€” it's just wrong for tags pushed by a workflow's own `GITHUB_TOKEN`). GitHub suppresses `on: push`/`on: create` etc. triggers for refs pushed using the DEFAULT `GITHUB_TOKEN`, specifically to prevent workflows from infinitely re-triggering each other â€” completely undocumented in the trigger's own syntax, only mentioned in GitHub's general Actions security docs. Symptom: tag existed, no error anywhere, no release ever appeared â€” a fully silent failure. Fix: use `workflow_call` (a direct reusable-workflow invocation, not an event) instead of relying on a push-triggered chain. **Pattern**: any time one workflow is meant to trigger another via a git push/API action performed BY that first workflow (not by a human), check GitHub's GITHUB_TOKEN trigger-suppression rules first â€” the "obvious" architecture (push a tag, let the existing tag-push trigger pick it up) silently doesn't work.
- **Auto-release: `github.event_name` inside a `workflow_call` is the CALLER's event, not `"workflow_call"`** â€” found immediately after fixing the bug above, by testing again: the fix shipped a real GitHub release tagged literally `"main"` (the branch name) and marked it **Latest**, briefly serving a broken `module.json` (version="main") on the public manifest URL every Foundry install checks for updates. Root cause: `release.yml`'s tag-detection step checked `github.event_name == "workflow_call"` to decide whether to use the `version` input or `$GITHUB_REF_NAME`; inside a called reusable workflow, `event_name` reflects the event that triggered the TOP-level run (here, `auto-release.yml`'s own `push` trigger), never `"workflow_call"` â€” so the check silently fell through to `$GITHUB_REF_NAME`, which for a push-to-branch event is the branch name. Fix: branch on whether `inputs.version` was actually supplied (non-empty) instead of inspecting `event_name` at all â€” works identically for `workflow_dispatch` and `workflow_call`, immune to which event kicked off the run graph. Required user sign-off to delete the broken release + its colliding `main`-named git tag (which had already interfered with a plain `git fetch origin main`) since deleting pushed refs/releases is a destructive, publicly-visible action outside standing authorization. **Pattern**: `github.event_name` and similar top-level `github.*` context fields do NOT reset per reusable-workflow call â€” they describe the outer run, not the immediate trigger; a reusable workflow that needs to know "was I called with an explicit input" should check the input itself, never infer it from `event_name`.
