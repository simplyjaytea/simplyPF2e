## 2026-07-17 — Focus-spell support (PC + NPC), doc restructuring

Roadmap item, previously listed unbuilt. Planned before coding rather than delegated blind:

- **Planning**: an Opus agent (`subagent_type: Plan`) did the real-source research and produced a two-part plan, correcting an assumption that would otherwise have shipped wrong — PC focus points can't be derived by just embedding real class items, because the actual grant mechanism (a Cleric's deity, a Sorcerer's bloodline, ...) is a `ChoiceSet`/`GrantItem` subclass-resolution chain this module doesn't have and isn't worth building for this feature. The plan's workaround: the AI names 1-3 focus spells directly (mirroring how every other real-content pick in this module works), grounded against the compendium by the `"focus"` trait (verified: real focus spells carry an EMPTY `system.traits.traditions`, so the existing tradition-filtered `getSpellCandidates()` can never match one).
- **Part A (PC)**, branch `feat/pc-focus-spells`, commit `03381c3`, built by fable: `ai.mjs` gains an optional `focusSpells` schema field (sibling to `spellcasting`, not nested — a Champion has focus spells with no slots at all); `pc-builder.mjs` clamps/caps at 3 and builds a `prepared:"focus"` spellcasting entry with no `slots` object; `rule-templates.mjs` gains a `focusPool` exemplar kind (clones a real `ActiveEffectLike` rule, e.g. Clarity of Focus's `{mode:"add", path:"system.resources.focus.max", value:1}` — never hand-authored, same safety principle as the item forge) because `character/document.ts` silently zeroes `resources.focus.max` every data-prep pass unless a real Rule Element rebuilds it; `builder.mjs` gains a shared `resolveFocusSpells()` used by both pipelines. `focusPool` is deliberately excluded from `EFFECT_KINDS` (the item-forge-facing kind list) via a new internal `ALL_KINDS`/`EFFECT_KINDS` split, so it doesn't leak into the wondrous-item AI schema as an offerable effect.
- **Part B (NPC)**, branch `feat/npc-focus-spells` (stacked on Part A), commit `3755715`, built by fable: same entry shape, but NPCs need NO Rule Element at all — `npc/document.ts` merges `system.resources.focus` with the actor's own source data winning, so `{value, max}` set directly on the actor's system data just persists. Gated on `concept.spellcasting` being present (a focus-only NPC with no casting tradition has no DC to cast from — explicitly out of scope for v1, not built), enforced twice (normalize-time force-to-`[]`, and again at resolve-time since `#refineSpells` can null `concept.spellcasting` after normalize already ran).
- **User sign-off on 3 judgment calls** (pool size, focus-only-NPC scope, AI-opt-in-vs-hardcoded-rule) — all three took the plan's recommended default. Reasoning preserved in code comments at each block (`builder.mjs`'s NPC focus block cites the same "defensible default, not GM-Core-verified" caveat already used for `TREASURE_BY_LEVEL`).
- **Verification**: 2 new pure-logic self-checks (`pc-builder.focusSpells.test.mjs`, `builder.focusSpells.test.mjs`) cover the cap-at-3 clamp and the NPC's spellcasting-gate defense-in-depth — all 7 self-checks (5 prior + these 2) pass, `node --check` clean on every touched file. Everything else (the cloned RE actually producing a nonzero pool max on a real sheet, the entry rendering as castable, `resources.focus` surviving data-prep in practice, whether the AI's focus-spell picks are actually sensible) is unverified without a live Foundry world.
- **Not merged.** Both branches sit stacked, unpushed, pending a live-test pass — this is a schema-dependent feature in exactly the shape that's bitten this module before (see the bug log below), so budget for at least one live-test bug round before trusting it.

## 2026-07-15 — PC generator: initial build, first bug-fix pass, round 2 (three separate sessions, same calendar day)

### Early: PR #49 — Player Character generator mode ships

Repo got renamed/migrated from `simplyjaytea/pf2eainpcgen` to `simplyjaytea/simplyPF2e` mid-session. Built and merged **PR #49**: an encounter-mode random-theme dice button, and a new **Player Character generator mode** (new files `scripts/pc-builder.mjs`, `scripts/pc-tables.mjs`; touched `ai.mjs`, `compendium.mjs`, `builder.mjs`, `generator-app.mjs`, `sources-app.mjs`, `generator.hbs`, `lang/en.json`).

Core design insight: PCs don't need a benchmark-table math layer like NPCs do — once real Ancestry/Background/Class/feat items are embedded on a `type: "character"` actor with the right `system.build` data, the PF2e system's own derived-data prep computes AC/HP/saves/proficiencies itself. Ancestry/Background/Class come from a free-text prompt (AI drafts names, a grounded pass matches them against real `pf2e.ancestries`/`pf2e.backgrounds`/`pf2e.classes`/`pf2e.heritages` packs — heritages needed their own 4th pack category, they don't live inside ancestries). Feat slots filled from `pf2e.feats-srd`, discriminated by the feat's real `system.category` field. Spells/equipment/loot reuse `#refineSpells`/`#refineEquipment`/`#refineLoot` and `resolveLoot`/`applyTreasureBudget` completely unchanged by normalizing the PC concept to carry the same fields those helpers already read. Starting wealth reads `T.TREASURE_BY_LEVEL.total` scaled by `TREASURE_AMOUNT_MULTIPLIER`, deliberately NOT `treasureBudget()` (wrong divisor — that one's for an NPC's per-encounter share). Single-class only, fully automatic, both explicit v1 scope decisions.

**Shipped with a real, acknowledged risk**: no running Foundry+pf2e world was available while building this, so the exact `system` field names `createCharacterActor()` writes were asserted from training knowledge, not verified against live data — marked with a `SCHEMA NOTE — ASSERTED, NOT VERIFIED` comment with exact verification steps, called out as PR #49's first test-plan checkbox. It merged anyway (see bug log below for what that cost).

Process note: implementation was delegated to a `fable`-model agent in an isolated worktree after codebase research + an approved plan were done directly; the agent independently found real refinements (the heritages-pack split, the `system.category` feat-filter field) not in the original plan.

### Mid: PR #54 — first live-test bug-fix pass

User tested PR #49 live and filed GitHub issues #50-#53. Root-caused #50 items 1-7 and #51 by pulling real `foundryvtt/pf2e` source instead of guessing again:

- **Attribute boosts**: the actor's `system.build.attributes.boosts` only holds the tier-level free boosts (1/5/10/15/20) — ancestry's 2 free boosts, background's 2 boosts, and the class's key-ability boost are NOT actor-level data at all. Each ABC item has its own `system.boosts[*].selected` (ancestry/background) or `system.keyAbility.selected` (class), and that item's own `prepareActorData()` pushes the selection into `actor.system.build.attributes.boosts.{ancestry,background,class}` during data prep. New `assignItemBoosts()`/`resolveKeyAbility()` fix it. Also fixed `build.attributes.manual` from `true` to `false` (manual mode skips boost application entirely).
- **Feats in Bonus feats instead of their real slot**: `feats/group.ts`'s `assignFeat()` requires `system.location` to equal the slot id (`"<group>-<level>"`, e.g. `"ancestry-1"`), not the bare group name. Fixed by setting `system.location` + `system.level.taken` from the resolved feat slot.
- **No spell slots**: `spellcastingEntry.system.slots` was never populated. New `spontaneousSpellSlots(level)` in `pc-tables.mjs`, spot-checked against the known Player Core table.
- **0 starting HP**: `system.attributes.hp.value` never set. `character/document.ts`'s HP prep does `stat.value = Math.min(stat.value, stat.max)` every pass — seeding `hp.value: 9999` reliably clamps to the real computed max.
- **Missing bio/appearance**: `CharacterBiography` has no `.value` field — real fields are `.backstory`/`.appearance`.
- **Skills not selected**: `class.system.trainedSkills.additional` (a free-pick count) was never filled; background Lore needs a real embedded `type:"lore"` item, not a `system.skills` entry.
- **Missing armor**: prompt-only fix — `pcSystemPrompt()` never mentioned armor at all.
- **#51 — background lookup aborts the whole generator**: `getPacksFor()` hardcoded exactly one pack id per category, missing add-on/Lost-Omens compendiums. New `getAllPacksFor(category)` unions configured + auto-detected packs.

Process note: planned + verified directly (pulled real pf2e source via `gh api`/`curl`), delegated implementation to an Opus subagent with those source files as ground truth, then independently re-verified the highest-risk part (the attribute-boost chain) rather than trusting the subagent's self-report. **Merged without the PR's own test plan being run** (live in-Foundry generation) — user asked to push+merge immediately.

## 2026-07-12 — Item forge Phase 3, loot/CI fixes, auto-release

- **Magic item forge Phase 3** (branch `feat/itemforge-phase3`, builds on Phase 1/2): rune-based magic weapons/armor. A "kind" select (Wondrous Item / Weapon / Armor) switches to a different, simpler pipeline. **Key discovery**: fundamental AND property runes are just real Item documents in the equipment compendium with their own real level and price — verified via `packs/pf2e/equipment/*.json`, `weapon/data.ts`, `armor/data.ts`. So a runed item needs no empirical price benchmark and no memorized rune list: pick real base weapon/armor + real rune items, SUM their real prices, overall level = MAX level among them. Candidate harvesting filters base items by type/level, property runes by their real `system.usage.value` string, fundamental tiers by resolving each tier's real item level against the target. **The one non-obvious transform**: `system.runes.property` stores rune keys in camelCase ("ghostTouch", "greaterFlaming"), not the kebab-case slug the item's name would normally produce — verified against `WEAPON_PROPERTY_RUNES`/`ARMOR_PROPERTY_RUNES` in `runes.ts`. `kebabToCamel(slugify(name))` bridges the two (see bug log for the graded-rune follow-up bug). Known gaps: no rune prerequisite/exclusivity checking, armor-category filtering is loose, shield/ammo runes out of scope.
- **Loot over-generation fix** (issue #28, branch `fix/loot-budget-and-scaling`): two independent causes. (1) Encounter mode gave EVERY role group the FULL per-encounter `treasureBudget()` instead of one shared budget split across groups — fixed by dividing by `members.length`. (2) The AI never saw the Stingy/Standard/Generous amount setting at all — `LOOT_GUIDE` became `lootGuide(amount)`, `SYSTEM_PROMPT` became `systemPrompt(amount)`, with a `LOOT_AMOUNT_GUIDE` map nudging item count/richness.
- **Auto-release CI** (branches `ci/auto-release-on-merge` → `fix/auto-release-workflow-call` → `fix/release-tag-detection`, 3 PRs to get right): every push to `main` auto-ships a release, bumping the last dot-segment of the latest tag. Two real bugs found only by merging and watching it run live — see bug log.
- **Item forge button discoverability** (issue #27): added a second, more visible entry point next to the Single/Encounter mode toggle in the generator window itself, calling the same `openItemForge()` singleton the sidebar button opens.
- **Treasure overpay fix** (branch `fix/treasure-per-level-overpay`): `ENCOUNTERS_PER_LEVEL` had been hardcoded to 4 as a guess; live play showed loot running ~3x too rich. Fixed to the GM-Core-derived `1000/80 = 12.5` (1,000 XP to level ÷ 80 XP per Moderate encounter for 4 players).

## 2026-07-11 — Treasure budget, item forge Phases 1-2, token-opt, prompt bug fixes

- **Spell selection**: 2-stage keyword-focus-then-pick (cuts unbounded spell-list token dump). **Equipment**: added a `value` field (gp estimate) end-to-end; unmatched equipment falls back to `customEquipmentItem()` instead of vanishing; prompt broadened to explicit adventuring-gear category, count 2-6→3-8. **AI image/portrait generation removed entirely** — bestiary-art matching (`findBestiaryArt`) is now the only art path. **Equipment grounding** (branch `feat/grounded-equipment-matching`): 1-stage pick-from-real-candidates, no AI focus call — keywords tokenized locally, `getEquipmentCandidates(level, keywords)` excludes treasure and caps item level at creature level.
- **Treasure budget** (branch `feat/treasure-budget`): module computes an expected gp value for loot, same authority split as AC/HP (tables own the numbers, AI only themes within them). `TREASURE_BY_LEVEL` (GM Core Table 10-9, per character level) ÷ `ENCOUNTERS_PER_LEVEL` (hardcoded to 4 at this point — see the 07-12 correction above) × `RARITY_TREASURE_MULTIPLIER` × `TREASURE_AMOUNT_MULTIPLIER` = `treasureBudget(level, rarity, amount)`. Design: encounter mode looks up PARTY level, not each member's own; each role group gets one constant treasure share split evenly across its current copies. Enforcement (`applyTreasureBudget`) flexes ONLY coin entries (±20% tolerance), never deletes named items. CAVEAT: table rows 13-20 were recalled with lower confidence than 1-12; -1/0/21-24 rows are extrapolated, not published — re-check against the book if high-level treasure feels off.
- **Magic item forge Phase 1** (branch `feat/magic-item-forge`): describe a wondrous item → real Item document with working passive Rule Elements. Safety principle established here and reused everywhere since: **clone real REs, never hand-author them** — `rule-templates.mjs` scans installed Item packs at runtime for published items already carrying each RE kind, exact-shape whitelists the rule, and `buildMagicItemData()` clones+parameterizes only value/selector/damage-type. 6 effect kinds shipped with runtime-found exemplars (itemBonus, resistance, weakness, immunity, sense, speed). Kinds with no exemplar in a given world are dropped from the AI's schema dynamically, never hand-authored as fallback. Pricing is empirical: median real compendium gp at the item level, window widens when `<5` samples.
- **Magic item forge Phase 2** (branch `feat/magic-item-forge-phase2`): 1/day ACTIVATED item effects via a click-to-run companion macro, not passive rules. **The AI never writes code** — it supplies only numbers/enum slugs, embedded as `JSON.stringify` constants atop a pre-written script body (zero injection surface). Four templates: damage, heal, condition, selfBuff. Every non-trivial API call is try/catch → plain-chat-message fallback. Several APIs (`actor.saves[slug].roll()`, DamageRoll-triggers-Apply-buttons behavior, `actor.increaseCondition()`, the `effect` item's duration shape) could NOT be verified against real system source on the machine available at the time — recalled-and-wrapped, not verified against ground truth; flagged as the first things to check if a macro misbehaves. Charges tracked via a module flag (`flags.simplypf2e.forge.uses`) rather than the system's native fields, since those couldn't be verified either.

## Bug log

Chronological, oldest first. The **Pattern:** lines are the actually-reusable lessons — read these before repeating an investigation.

- **PC generator shipped with 7+ schema field-name/behavior bugs from an unverified assertion** (fixed, PR #54) — the PC generator (PR #49) had a `SCHEMA NOTE — ASSERTED, NOT VERIFIED` comment acknowledging its `system.*` field names were recalled from training knowledge, not checked against a live Foundry+pf2e world. It merged anyway (auto-release ships every main merge). User tested live and every guessed field either had the wrong name or was missing a required companion field (see the PR #54 write-up above for the full list). **Pattern**: "no live Foundry world available" is not a reason to ship a guessed schema — the real system source is public on GitHub and can be pulled and read directly; do this BEFORE writing schema-dependent code, not after a user reports it's broken. A hedge comment ("asserted, not verified") is not a substitute for verification, it's just an honest label on a bug that hasn't been found yet.
- **PC generator's "verified against real source" fix round only verified the fields it touched — untouched fields were still guessed-and-wrong or simply never written** (fixed across PRs #57/#59/#61) — PR #54 fixed the 7 bugs the user had actually found and reported. A second live test (issue #56) found 6 MORE gaps in fields PR #54 never touched at all, and a THIRD pass (PR #61) found a background/ancestry/class's own automatically-granted feat landing in Bonus Feats for a different reason than the AI-picked-feat bug PR #54 fixed. **Pattern**: "verified against real source" is a claim about the fields that were checked, not the whole feature — a schema this large has enough surface area that fixing reported bugs doesn't mean unreported ones don't exist. When a feature has shipped with schema guesses even once, budget for at least one more live-test round finding fields nobody thought to check yet.
- **`ABCItemPF2e`'s own granted feats need a DIFFERENT `system.location` value than AI-picked feats** (fixed, PR #61) — PR #54's fix (`system.location = "<group>-<level>"`) is for feats filling a level-gated SLOT and was verified via `feats/group.ts`. But a background's own built-in feat (granted via `system.items`) needs `system.location` set to the GRANTING ITEM'S OWN embedded `_id` instead, per `ABCItemPF2e.createGrantedItems()`, matched against a hidden slot the Skill Feats group builds specifically for it. Two structurally different feat-placement mechanisms writing to the same field. **Pattern**: when a real system has more than one path that writes to the same field, verifying ONE path's mechanism doesn't mean a DIFFERENT path works the same way — pull the source for each mechanism separately rather than assuming they share a rule because they share a field name.
- **PC starting wealth used the wrong table row for level 1** (fixed, PR #61) — `pcStartingWealthGp()` fed every level, including 1, through `TREASURE_BY_LEVEL`, giving a fresh 1st-level character ~175 gp instead of the Core Rulebook's flat 15 gp. Caught only because a NEW feature (wealth-buys-items) made the bug's effect much more visible. **Pattern**: a benchmark-reuse decision ("this table already has the number I need") should be checked for documented EXCEPTIONS to the table's own applicability, not just for whether the number itself is right.
- **Attack of Opportunity not generating** (fixed, unreleased). Schema comment cited it as a valid glossary-ability example, but the design-guidance enumeration bullet never listed it, so the model almost never picked it. **Pattern**: check whether X is only a schema-comment example vs. actually reinforced in the "Design guidance" section — if schema-only, add it to the relevant enumeration/road-map bullet there too.
- **Gold coins appearing in equipment** (fixed, unreleased). The `equipment` schema comment never told the model not to put coins there, and equipment has no coin-to-currency conversion. Fixed with both a prompt note AND a defensive filter reusing `parseCoins()` to drop any equipment entry that parses as pure coins. **Pattern**: prompt-only fixes are necessary but not sufficient for "wrong bucket" bugs — pair with a cheap defensive filter using logic that already exists elsewhere, so a prompt regression doesn't silently reopen the bug.
- **Pre-Remaster item names leaking through** (fixed, `fix/remaster-item-names`, commit `8b29a94`). `LOOT_GUIDE` already reminded the model to use Remaster names; the `equipment` schema comment never had that reminder at all. Fixed via a shared `REMASTER_NOTE` const interpolated into both. **Pattern**: whenever the model names published content in more than one schema field, grep for where the naming-convention reminder already exists and make sure every sibling field gets it too — ideally via a shared const, not copy-pasted text.
- **Passives were inert prose instead of "just working"** (same commit). Custom (non-glossary-matched) passive abilities only get automation from `enrichDescription()`'s phrasing conventions — anything else is just prose a GM has to remember to apply. Fix (prompt-only, deliberately not a new RE-generation feature): nudged the model to prefer a real glossary match for passives, and to phrase necessarily-custom passives using the existing description conventions.
- **Encounter mode created one actor per GROUP, ignoring `member.count`** (fixed, `fix/encounter-actor-count-and-loot-scaling`). `composeEncounter()` correctly returns groups with counts, but `#createEncounterActors()` called `createActor()` once per member regardless. **Pattern**: when a data structure carries a count/quantity field, grep every consumer to confirm it actually multiplies by it.
- **Loot never scaled up for "hoard"/"lots of loot" requests** (fixed, same branch). Three independent caps each silently flattened abundance: a hardcoded "3-8 items" prompt baseline with no scaling instruction, `generateLoot()`'s context dropping `concept.description` (where hoard language actually lived), and `normalizeLoot()` hard-truncating to 12 entries. **Pattern**: a "model won't do X" bug can have BOTH a prompt cause and a post-processing cause — trace the output through every normalize/clamp step after fixing the prompt, and check the model can even SEE the signal (a token-trimmed context may have dropped the field the instruction keys on).
- **Treasure group budget multiplied by headcount instead of splitting across it** (caught pre-merge, `feat/treasure-budget`). First implementation set the per-copy share to the group's FULL target and multiplied by count for the readout — a 4-copy group got 4x intended treasure. Fixed to `treasureGroupBudget ÷ count`. **Pattern**: a per-group total and a per-copy share are easy to conflate; when a UI stepper can change a count live, check whether the derived value needs to be constant (recompute the split) or already-final.
- **Condition-activation macro would silently auto-apply on an unreadable save result** (fixed, `feat/magic-item-forge-phase2`, found by independent review). If the guessed save-result field shape was wrong, the "did it apply" check defaulted to `true` — silently turning a save-negates condition into an unconditional hit, no error, no warning. Fixed to default to the SAFER outcome (skip + manual-adjudication message) when the result can't be read. **Pattern**: when wrapping an uncertain external API in a boolean gate, make sure the "I couldn't read the result" branch defaults to the safer/weaker outcome — silent over-application is worse than silent under-application, especially for anything with real game-balance stakes.
- **Graded property runes ("Flaming (Greater)") keyed wrong in `system.runes.property`** (found by an independent `fable` audit, fixed after the original PR had already merged). `kebabToCamel(slugify(name))` turned "Flaming (Greater)" into `"flamingGreater"`, but the real key is `"greaterFlaming"` — the grade moves to a LEADING prefix. Would have silently failed to apply the rune's effect — no error, just inert. Fixed by special-casing the `"X (Greater|Major)"` shape and reordering before camelCasing, verified against 4 independent real examples. **Pattern**: when a transform is derived from a FEW verified samples, actively hunt for a shape that breaks the naive version before shipping.
- **Fundamental runes leaking into the property-rune candidate list** (same audit, same fix commit). Weapon/Armor Potency and Striking/Resilient share the exact same `system.usage.value` filter string as actual property runes, so they could be offered to and picked by the AI as a property rune, double-pricing the item. Fixed by explicitly excluding the fundamental catalog names. **Pattern**: when two categories of real compendium item share a filter field, don't assume the filter alone disambiguates them.
- **Auto-release: pushed tag never triggered the release build** (found by testing live immediately after merging). GitHub suppresses `on: push` triggers for refs pushed using the DEFAULT `GITHUB_TOKEN`, to prevent workflows infinitely re-triggering each other — undocumented in the trigger's own syntax. Symptom: tag existed, no error, no release, fully silent. Fixed via `workflow_call` (direct reusable-workflow invocation) instead of relying on a push-triggered chain. **Pattern**: any time one workflow is meant to trigger another via a git push/API action performed BY that first workflow, check GitHub's `GITHUB_TOKEN` trigger-suppression rules first.
- **Auto-release: `github.event_name` inside a `workflow_call` is the CALLER's event, not `"workflow_call"`** (found immediately after fixing the bug above). The fix shipped a real GitHub release tagged literally `"main"`, marked Latest, briefly serving a broken `module.json` on the public manifest URL. Root cause: the tag-detection step checked `event_name == "workflow_call"`, which inside a called reusable workflow reflects the TOP-level run's event instead. Fixed by branching on whether `inputs.version` was actually supplied, instead of inspecting `event_name` at all. Required user sign-off to delete the broken release + colliding git tag. **Pattern**: `github.event_name` and similar top-level `github.*` context fields do NOT reset per reusable-workflow call — a reusable workflow that needs to know "was I called with an explicit input" should check the input itself, never infer it from `event_name`.

## Process notes worth remembering generally

- The user tends to merge PRs themselves right after requesting a review/audit, sometimes before the audit finishes — say so explicitly if a requested review is still in flight rather than assuming it's understood; a merge landing mid-audit needs its fixes shipped as a new follow-up PR.
- CI/workflow (`.yml`) changes specifically need a live test merge to trust — `node --check`/YAML-syntax-validity has never caught a trigger-chain bug in this repo's history (2 for 2 so far).
- An independent second agent auditing a freshly-built diff (not just the builder's own self-report) has found real bugs on at least 3 separate occasions (item forge Phase 2's condition macro, item forge Phase 3's rune-key + fundamental-leak bugs). Worth doing for anything schema-dependent or security/balance-sensitive, even after the primary build already looks clean.
